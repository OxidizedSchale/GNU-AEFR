## 🚀 はじめに

>"Release? Real hackers compile from source." ;-)

[**Releases**](https://github.com/OxidizedSchale/GNU-AEFR/releases) ページにアクセスしてソースコードをダウンロードするか、お使いのプラットフォーム向けにコンパイル済みのバイナリ（命令セットの最適化なし）を直接取得してください。

GNU:AEFRは、**グラフィカルインターフェース (GUI)** と **コマンド駆動 (Command-Driven)** を組み合わせたインタラクション方式を採用しています。

*   **デスクトップ版**: グラフィカルインターフェースの使用を推奨します。
*   **モバイル版**: ファイルのインポートに関しては、現在コマンドのみ使用可能です。

画面左上の `[CMD]` ボタンをクリックすると、組み込みのデバッグコンソールが開きます（GUIサポートあり）。

## 📖 コマンドリファレンスマニュアル

### 1. シーンとビジュアル

*   **背景の読み込み**
    *   **コマンド**: `BG <image_path>`
    *   **説明**: 背景画像を瞬時に切り替えます。`.jpg`, `.png`, `.webp` をサポート。
    *   **例**: `BG C:\Assets\BlueArchive\BG_Classroom.png`

*   **キャラクターの装填**
    *   **コマンド**: `LOAD <slot_ID> <.atlas_path>`
    *   **説明**: キャラクターを `0` から `4` までの5つのスロットに読み込み、自動レイアウトをサポートします。読み込みに成功すると、コンソールに利用可能なアクションのリストが出力されます。
    *   **例**: `LOAD 0 D:\Assets\Shiroko\Shiroko_Home.atlas`

### 2. モーションと演出

*   **アクションの切り替え**
    *   **コマンド**: `ANIM <slot_ID> <animation_name> [loop: true/false]`
    *   **説明**: `true` はループ再生、`false` は1回のみ再生。アクション名は完全に一致する必要があります（キャラクター読み込み時にコンソールに利用可能なアクションがリストアップされます）。
    *   **例**:
        ```bash
        ANIM 0 Start_Idle_01 true    # シロコの待機アクションをループ再生させる
        ANIM 1 Attack_Normal false   # 1番スロットのキャラクターに1回攻撃させる
        ```

### 3. スクリプトとダイアログ

*   **ダイアログの送信**
    *   **コマンド**: `TALK <name>|<affiliation>|<content>`
    *   **説明**: 標準的なキヴォトススタイルのダイアログボックスをレンダリングし、タイプライターエフェクトをサポートします（ダイアログボックスをクリックすると瞬時にスキップ可能）。**パラメータは縦棒 `|` で区切る必要があります。**
    *   **例**:
        ```bash
        TALK 砂狼シロコ|対策委員会|先生、銀行強盗に行きますか？
        TALK アロナ|シッテムの箱|先生、仕事中にサボらないでください！
        ```

### 4. オーディオシステム

*   **BGM の再生**
    *   **コマンド**: `BGM <audio_path>`
    *   **説明**: BGMを非同期で読み込んで再生し、シームレスな切り替えをサポートします。
    *   **例**: `BGM D:\Music\Unwelcome_School.mp3`

*   **音楽の停止**
    *   **コマンド**: `STOP`
    *   **説明**: 現在再生中のBGMを即座に停止します。

### 💡 ギーク向けTips

*   **パスの問題**: Windowsではファイルパスを直接貼り付けることをお勧めします。AEFRが自動的に引用符（例: `"C:\Path"`）を処理します。Android / Termux では、絶対パス（例: `/sdcard/Download/bg.png`）を使用してください。
*   **パフォーマンスモニタリング**: 「ジェントルマンスケジューラ」のおかげで、5つのスロットをすべて埋めてBGMを再生していても、UIスレッドは滑らかなままです。大胆にマルチタスクを行ってください。
*   **ログフィードバック**: すべての操作結果（成功/失敗/利用可能なアニメーションリスト）は、コンソールのLogエリアにリアルタイムで出力されます。必ず確認してください。

### 技術スタックの純潔性
本プロジェクトは、コアのビジネスロジックとアーキテクチャの100%を **Rust** で実装することにこだわっています。

*   **原則として**、C++ランタイムやフレームワーク（Qt、Unity、Unrealなど）との複雑なインタラクション（JNI/複雑なFFI）を導入するPRは、アーキテクチャの純粋性と保守性を維持するために**拒否**します。
*   **例外**: 既存の、機能が完全に揃った、低レイヤーのシステムレベルCライブラリ（グラフィックス、オーディオ、ファイルシステムなど）を統合するための、安全なRustラッパーの作成は許可されます。その場合：
    *   コミュニティでメンテナンスされている成熟した `-sys` バインディング（例: `libc`, `openssl-sys`）を優先しなければなりません。
    *   FFI呼び出しを行うために自身で `unsafe` コードを記述する必要がある場合は、以下の `unsafe` コードの基準を厳密に遵守し、その代替不可能性を証明しなければなりません。
    *   最終的な目標は、すべての `unsafe` 呼び出しを安全なRust API内にカプセル化し、上位アプリケーションに対して完全に透過的にすることです。

### unsafe ルール：安全性の上にあるパフォーマンスの刃
> `unsafe` キーワードは、コンパイル時の安全ルールを超えて必要な低レイヤー操作を行う能力をRustが開発者に与えるものです。本プロジェクトではこれを「安全性の上にあるパフォーマンスの刃」と見なし、その使用原則は「必要でなければ使用しない。使用する場合は万全を期す」です。

**コア原則：必要性の証明**

いかなる `unsafe` コードブロックの存在も、安全なRustコードでは実現できないという正当な理由に基づいている必要があります。PRを提出する際、`unsafe` コードブロックのすぐ上に、簡潔な「必要性の証明」をコメントとして提供しなければなりません。これには少なくとも以下を含める必要があります：

1.  **理由**: なぜ `unsafe` を使用する必要があるのかを明確に示します（例：特定のC FFI関数の呼び出し、確定的なメモリレイアウトの変換、自己参照データ構造の実装など）。
2.  **代替不可能性**: なぜ安全なRust標準ライブラリやコミュニティライブラリで同じ機能を実現できないのかを論証します。
3.  **安全の境界**: その `unsafe` ブロックが維持することを約束する不変条件を明確に定義します。つまり、開発者としてコンパイラにどのような条件が「必ず成り立つ」と約束し、そのコードを論理的に安全なものにしているのかを明記します。

*具体的な要件：ドキュメント化されたコメント*
*   すべての `unsafe` 関数、メソッド、またはコードブロックには、必ず前置きのコメントを付けなければなりません。

</details>

## 🏛️ アーキテクチャの哲学

<details>
<summary><strong>クリックして展開：GNU:AEFRを駆動するハードコアな低レイヤー設計について理解する</strong></summary>

### ジェントルマンスケジューラ（Gentleman Scheduler）：演算能力レベルの階級隔離
> GNU:AEFR v0.8+ では、独自開発の「ジェントルマンスケジューラ」を導入しました。私たちはオペレーティングシステムのデフォルトのスケジューリングを信用していません。なぜなら、それらはしばしば「公平性」という名目のために、リアルタイムレンダリングの確実性を犠牲にするからです。

*   **コア基準：N-2 戦略**
    *   GNU:AEFRはCPUを食い尽くす強欲な獣になることを拒否します。スケジューラは物理コア数 `N` を自動的に検出し、強制的に `N-2` 個のコアを計算専用領域として隔離します（デュアルコアの場合は1コアを占有します）。
    *   **1 コア** を UI/Render (Main Thread) 用に予約：計算量が急増した場合でも、レンダリングのフレームレートが滑らかであることを保証します。
    *   **1 コア** を OS/Audio (Backstage) 用に予約：底層システムの「ショックアブソーバー（緩衝地帯）」として機能し、CPUがフルロードになった際にBGMに音割れやノイズが発生するのを完全に防ぎます。
    *   残りのコアを GNU:AEFR Worker に割り当て：`Rayon` スレッドプールを利用して Spine のスケルトンスキニングと物理計算を行います。

*   **なぜ「同期ブロックモデル」にこだわるのか？**
    > Web開発でよく見られる安価な「ノンブロッキング非同期」の概念を私たちに売り込まないでください。GNU:AEFRの哲学において、「同期」とは秩序です。

    *   **画面のティアリングを拒否**: メインスレッドは Update フェーズ中に計算結果を同期的に待機します。これは、各フレームのスケルトンの位置、表情、物理効果が空間的に完全に整列していることを保証するためです（立ち絵が乱れ飛ぶのは見たくないでしょう）。
    *   **Work Stealing（ワークスティーリング）機構**: Rayonのハードコアな実装のおかげで、コア数が少ない極端な場合（デュアルコアマシンなど）には、メインスレッドが自ら計算に参加し、計算能力の利用率が100%の物理的限界に達するようにします。
    *   **スケジューラの警告**: いかなる計算集約型タスクにおいても `std::thread::spawn` の使用を禁止します。すべての並行タスクは `AefrScheduler` を介してディスパッチされなければなりません。違反した場合、システムスケジューリングが無秩序な競争に陥り、これはGNU:AEFRのRustの美学に対する裏切りとなります。

### GUI 改善ルール
*   UIコンポーネントは `egui` の**イミディエイトモード (Immediate Mode)** の哲学に厳密に従う必要があり、`React` スタイルの提案は受け付けられません。
