## 🚀 はじめに

> "Release? Real hackers compile from source." ;-)

[**Releases**](https://github.com/OxidizedSchale/GNA-AEFR/releases) ページからソースコードをダウンロードするか、お使いのプラットフォーム向けにコンパイルされたバイナリを直接取得してください。

**GNA:AEFR** は、**GUI（グラフィカルユーザーインタフェース）**と**コマンド駆動（Command-Driven）**を組み合わせたインタラクション方式を採用しています。

*   **デスクトップ版**: GUIの使用を推奨します。
*   **モバイル版**: ファイルのインポートに関しては、現在コマンド入力のみ対応しています。

画面左上の `[CMD]` ボタンをクリックすると、内蔵デバッグコンソール（GUIサポート付き）が開きます。

## 📖 コマンドリファレンスマニュアル

### 1. シーンとビジュアル

*   **背景の読み込み**
    *   **コマンド**: `BG <image_path>`
    *   **説明**: 背景画像を瞬時に切り替えます。`.jpg`, `.png`, `.webp` をサポート。
    *   **例**: `BG C:\Assets\BlueArchive\BG_Classroom.png`

*   **キャラクターの装填**
    *   **コマンド**: `LOAD <slot_ID> <.atlas_path>`
    *   **説明**: キャラクターを `0` から `4` までの計5つのスロットに読み込みます。自動レイアウトをサポート。読み込み成功後、コンソールに使用可能なモーションの一覧が表示されます。
    *   **例**: `LOAD 0 D:\Assets\Shiroko\Shiroko_Home.atlas`

### 2. モーションと演出

*   **モーションの切り替え**
    *   **コマンド**: `ANIM <slot_ID> <animation_name> [loop: true/false]`
    *   **説明**: `true` でループ再生、`false` で1回再生。モーション名は正確に一致する必要があります（読み込み時にコンソールに表示されるリストを参照してください）。
    *   **例**:
        ```bash
        ANIM 0 Start_Idle_01 true    # シロコの待機モーションをループ再生
        ANIM 1 Attack_Normal false   # 1番スロットのキャラが1回攻撃
        ```

### 3. スクリプトと対話

*   **ダイアログの送信**
    *   **コマンド**: `TALK <name>|<affiliation>|<content>`
    *   **説明**: 標準的なキヴォトススタイルのダイアログボックスをレンダリングします。タイプライター効果をサポート（クリックでスキップ可能）。**パラメータは必ず垂直バー `|` で区切ってください。**
    *   **例**:
        ```bash
        TALK 砂狼シロコ|対策委員会|先生、銀行強盗に行きますか？
        TALK アロナ|シッテムの箱|先生、仕事中にサボらないでください！
        ```

### 4. オーディオシステム

*   **BGMの再生**
    *   **コマンド**: `BGM <audio_path>`
    *   **説明**: BGMを非同期で読み込み、シームレスに切り替えて再生します。
    *   **例**: `BGM D:\Music\Unwelcome_School.mp3`

*   **音楽の停止**
    *   **コマンド**: `STOP`
    *   **説明**: 現在再生中のBGMを直ちに停止します。

---

### 💡 ギーク向けチップス

*   **パスの問題**: Windowsではファイルパスを直接コピーすることを推奨します。AEFRは引用符を自動的に処理します（例: `"C:\Path"`）。Android / Termuxでは、 `/sdcard/Download/bg.png` のような絶対パスを使用してください。
*   **パフォーマンス監視**: 「ジェントルマンスケジューラ」のおかげで、5つのスロットをすべて埋めてBGMを再生しても、UIスレッドは滑らかなままです。安心してマルチタスクを行ってください。
*   **ログフィードバック**: すべて操作結果（成功/失敗/利用可能なモーション一覧）は、コンソールのLogエリアにリアルタイムで表示されます。

### 技術スタックの純粋性
本プロジェクトは、コアロジックとアーキテクチャの100%を**Rust**で実装することにこだわっています。

*   **C++インターオピの拒絶**: アーキテクチャの純粋性と保守性を維持するため、C++ランタイムやフレームワーク（Qt, Unity, Unrealなど）との複雑なやり取り（JNI/複雑なFFI）を導入するPRは原則として拒否します。
*   **例外事項**: 既存の機能が完成された低レイヤCライブラリ（グラフィックス、オーディオ、ファイルシステム基礎ライブラリ）にアクセスするためのRustセーフラッパーの作成は許可されます。その場合：
    *   コミュニティで保守されている成熟した `-sys` バインディングライブラリを優先的に使用すること。
    *   FFI呼び出しのために `unsafe` コードを書く必要がある場合は、後述の `unsafe` 規範を厳守し、その不可代替性を証明すること。
    *   最終的な目標は、すべての `unsafe` 呼び出しを安全なRust API内にカプセル化し、上位層のアプリケーションに対して完全に透過的にすることです。

### unsafe 規範：安全の上にある性能の刃
> `unsafe` キーワードは、コンパイラの安全ルールの外で必要な低レイヤ操作を行うためにRustが開発者に与えた能力です。本プロジェクトでは、これを「安全の上にある性能の刃」と見なし、その使用原則を「必要不可欠でない限り使用せず、使用する場合は万全を期す」としています。

**核心原則：必要性の証明**

いかなる `unsafe` ブロックも、安全なRustコードでは実現不可能な正当な理由に基づいている必要があります。PRを提出する際は、 `unsafe` ブロックの直前にコメント形式で簡潔な「必要性の証明」を提供してください。内容は少なくとも以下を含む必要があります：

1.  **理由**: なぜ `unsafe` を使用しなければならないのかを明記（例：特定のC FFI関数の呼び出し、確定的なメモリレイアウト変換、自己参照構造体の実装など）。
2.  **不可代替性**: なぜ標準ライブラリやコミュニティのライブラリで同じ機能を実現できないのかを論証。
3.  **安全境界**: その `unsafe` ブロックが維持することを約束する不変条件（Invariants）を明確に定義。つまり、開発者として、どのような条件が成立することをコンパイラに「約束」し、そのコードを論理的に安全にしているのか。

*具体的要件：ドキュメント化コメント*
*   すべての `unsafe` 関数、メソッド、またはコードブロックには、Safety注釈を付与しなければなりません。

## 🏛️ アーキテクチャ哲学

<details>
<summary><strong>クリックして展開：GNA:AEFRを駆動するハードコアな低レイヤ設計</strong></summary>

### ジェントルマンスケジューラ：演算レベルでの階級隔離
> GNA:AEFR v0.8+ では、独自開発の「ジェントルマンスケジューラ」を導入しました。私たちはOSのデフォルトのスケジューリングを信頼していません。なぜなら、それらはしばしば「公平性」のためにリアルタイムレンダリングの確定性を犠牲にするからです。

*   **核心規範：N-2 戦略**
    *   GNA:AEFRはCPUを食いつぶす強欲な野獣になることを拒みます。スケジューラは物理コア数 `N` を自動検出し、 `N-2` 個のコアを計算専用ゾーンとして強制的に隔離します（コア/スレッドが2つしかない場合は、AEFRが1つを占有します）。
    *   **1 コア** を UI/Render (Main Thread) 用に予約：計算量が急増しても、レンダリングフレームレートが滑らかに保たれるようにします。
    *   **1 コア** を OS/Audio (Backstage) 用に予約：低レイヤシステムの「緩衝地帯」として、CPUフルロードによるBGMのノイズや途切れを徹底的に排除します。
    *   残りのコアを **GNA:AEFR** Worker に割り当て：Rayonスレッドプールを利用して、Spineのボーンスキニングと物理演算を行います。

*   **なぜ「同期ブロッキングモデル」に固執するのか？**
    > Web開発で見られるような安価な「非ブロッキング非同期」の概念を私たちに押し付けないでください。GNA:AEFRの哲学において、「同期」とは「秩序」です。

    *   **画面引き裂き（ティアリング）の拒絶**: メインスレッドはUpdateフェーズで計算結果を同期的に待ちます。これは、毎フレームのボーン位置、表情、物理効果が空間的に完全に一致していることを保証するためです。
    *   **ワークスティーリング（Work Stealing）メカニズム**: Rayonのハードコアな実装により、コア数が少ない極端な状況（デュアルコアマシンなど）でも、メインスレッドが自ら計算に参加し、演算能力の利用率を物理的な限界である100%まで高めます。
    *   **スケジューラ警告**: 計算集約型タスクで `std::thread::spawn` を使用することを禁止します。すべての並列タスクは `AefrScheduler` を通じて配布されなければなりません。これに違反すると、システムスケジューリングが無秩序な競争に陥り、それは GNA:AEFR のRust美学に対する裏切りとなります。

### GUI 改善ルール
*   UIコンポーネントは `egui` の**イミディエイト・モード（即時モード）**哲学に従わなければなりません。`React` スタイルの宣言的な提案は受け入れられません。
