# GNU's Not Unix : AEFR's Eternal Freedom & Rust-rendered

> GNU에서 영감을 받았지만, 공식 GNU 프로젝트는 아닙니다. '자유 소프트웨어 재단(FSF) 자유 소프트웨어 디렉토리'에 등재를 신청했습니다.

## 우리의 Rust 순도는 GNU/Linux 커널보다 99.7% 더 높습니다!

**GNU:AEFR은 GNU 선언의 정신을 계승하는 자유 소프트웨어 프로젝트이며, 키보토스(Kivotos)의 창작 환경을 해방하기 위해 헌신하고 있습니다! 소프트웨어를 오픈소스로 공개하지 않고 배포하는 것은 무책임한 행동입니다!**

## 📖 명령어 참고 매뉴얼

### 1. 씬 및 비주얼

*   **배경 로드**
    *   **명령어**: `BG <image_path>`
    *   **설명**: 배경 이미지를 즉시 전환합니다. `.jpg`, `.png`, `.webp` 지원.
    *   **예시**: `BG C:\Assets\BlueArchive\BG_Classroom.png`

*   **캐릭터 장착**
    *   **명령어**: `LOAD <slot_ID> <.atlas_path>`
    *   **설명**: 캐릭터를 `0`번부터 `4`번까지 총 5개의 슬롯에 로드하며, 자동 레이아웃을 지원합니다. 로드에 성공하면 콘솔에 해당 캐릭터의 사용 가능한 동작 목록이 출력됩니다.
    *   **예시**: `LOAD 0 D:\Assets\Shiroko\Shiroko_Home.atlas`

### 2. 모션 및 연출

*   **동작 전환**
    *   **명령어**: `ANIM <slot_ID> <animation_name>[loop: true/false]`
    *   **설명**: `true`는 루프(반복) 재생, `false`는 한 번 재생합니다. 동작 이름은 정확히 일치해야 합니다 (캐릭터 로드 시 콘솔에 사용 가능한 동작이 나열됨).
    *   **예시**:
        ```bash
        ANIM 0 Start_Idle_01 true    # 시로코의 대기 모션을 루프 재생시킴
        ANIM 1 Attack_Normal false   # 1번 슬롯의 캐릭터가 한 번 공격하도록 함
        ```

### 3. 스크립 및 대화

*   **대화 전송**
    *   **명령어**: `TALK <name>|<affiliation>|<content>`
    *   **설명**: 표준 키보토스 스타일의 대화 상자를 렌더링하며, 타자기 효과를 지원합니다 (대화 상자를 클릭하면 즉시 스킵 가능). **매개변수는 반드시 수직선 `|`으로 구분해야 합니다.**
    *   **예시**:
        ```bash
        TALK 스나오오카미 시로코|대책위원회|선생님, 은행 털러 갈까요?
        TALK 아로나|싯팀의 상자|선생님, 근무 시간에 농땡이 피우지 마세요!
        ```

### 4. 오디오 시스템

*   **BGM 재생**
    *   **명령어**: `BGM <audio_path>`
    *   **설명**: 배경 음악을 비동기적으로 로드하고 재생하며, 매끄러운 전환을 지원합니다.
    *   **예시**: `BGM D:\Music\Unwelcome_School.mp3`

*   **음악 정지**
    *   **명령어**: `STOP`
    *   **설명**: 현재 재생 중인 BGM을 즉시 정지합니다.

### 기술 스택의 순수성
본 프로젝트는 핵심 비즈니스 로직과 아키텍처의 100%를 **Rust**로 구현하는 것을 고집합니다.

*   **원칙적으로**, 아키텍처의 순수성과 유지보수성을 위해 C++ 런타임이나 프레임워크(Qt, Unity, Unreal 등)와의 복잡한 상호 작용(JNI/복잡한 FFI)을 도입하는 PR은 **거부**합니다.
*   **예외 상황**: 기존의 완전한 기능을 갖춘 로우 레벨 시스템 C 라이브러리(그래픽, 오디오, 파일 시스템 등)를 통합하기 위한 안전한 Rust 래퍼(Wrapper) 작성은 허용됩니다. 이 경우:
    *   커뮤니티에서 유지보수하는 성숙한 `-sys` 바인딩(예: `libc`, `openssl-sys`)을 우선적으로 사용해야 합니다.
    *   FFI 호출을 위해 직접 `unsafe` 코드를 작성해야 한다면, 아래의 `unsafe` 코드 기준을 엄격히 준수하고 그 대체 불가능성을 증명해야 합니다.
    *   최종 목표는 모든 `unsafe` 호출을 안전한 Rust API 내에 캡슐화하여 상위 애플리케이션에 완전히 투명하게 만드는 것입니다.

### unsafe 규칙: 안전 위에 있는 성능의 칼날
> `unsafe` 키워드는 컴파일 타임의 안전 규칙을 넘어서 필요한 로우 레벨 작업을 수행할 수 있도록 Rust가 개발자에게 부여한 능력입니다. 본 프로젝트는 이를 "안전 위에 있는 성능의 칼날"로 간주하며, 그 사용 원칙은 "필요하지 않으면 사용하지 않되, 사용할 때는 완벽을 기한다"입니다.

**핵심 원칙: 필요성 증명**

모든 `unsafe` 코드 블록은 안전한 Rust 코드로 구현할 수 없는 정당한 이유를 기반으로 해야 합니다. PR 제출 시, `unsafe` 코드 블록 바로 위에 짧은 "필요성 증명"을 주석으로 제공해야 하며, 여기에는 최소한 다음 내용이 포함되어야 합니다:

1.  **이유**: 왜 `unsafe`를 사용해야만 하는지 명확히 밝힙니다 (예: 특정 C FFI 함수 호출, 확정적인 메모리 레이아웃 변환, 자기 참조 데이터 구조 구현 등).
2.  **대체 불가능성**: 왜 안전한 Rust 표준 라이브러리나 커뮤니티 라이브러리로 동일한 기능을 구현할 수 없는지 논증합니다.
3.  **안전 경계**: 해당 `unsafe` 블록이 유지하기로 약속하는 불변 조건(Invariants)을 명확히 정의합니다. 즉, 개발자로서 컴파일러에게 어떤 조건이 "반드시 성립한다"고 약속하여 이 코드를 논리적으로 안전하게 만드는지 명시합니다.

*구체적 요구 사항: 문서화된 주석*
*   모든 `unsafe` 함수, 메서드 또는 코드 블록에는 반드시 선행 주석을 달아야 합니다.

<summary><strong>클릭하여 펼치기: GNU:AEFR을 구동하는 하드코어 로우 레벨 설계 이해하기</strong></summary>

### 젠틀맨 스케줄러 (Gentleman Scheduler): 연산 능력 수준의 계급 격리
> GNU:AEFR v0.8+에서는 자체 개발한 "젠틀맨 스케줄러"를 도입했습니다. 우리는 운영체제의 기본 스케줄링을 신뢰하지 않습니다. 왜냐하면 그것들은 종종 소위 "공정성"을 위해 실시간 렌더링의 확실성을 희생하기 때문입니다.

*   **핵심 기준: N-2 전략**
    *   GNU:AEFR은 CPU를 집어삼키는 탐욕스러운 짐승이 되기를 거부합니다. 스케줄러는 물리적 코어 수 `N`을 자동으로 감지하고, 강제로 `N-2`개의 코어를 연산 전용 구역으로 격리합니다 (듀얼 코어의 경우 1개의 코어 차지).
    *   **1 코어**를 UI/Render (Main Thread)에 예약: 연산량이 폭증할 때도 렌더링 프레임 속도가 매끄럽게 유지되도록 보장합니다.
    *   **1 코어**를 OS/Audio (Backstage)에 예약: 로우 레벨 시스템의 "완충 공간" 역할을 하여, CPU 풀 로드 시 BGM에 노이즈나 끊김이 발생하는 것을 완전히 차단합니다.
    *   나머지 코어를 GNU:AEFR Worker에 할당: `Rayon` 스레드 풀을 활용하여 Spine 스켈레탈 스키닝 및 물리 연산을 수행합니다.

*   **왜 "동기식 블로킹 모델"을 고집하는가?**
    > 웹 개발에서 흔히 볼 수 있는 값싼 "논블로킹 비동기" 개념을 우리에게 권유하지 마십시오. GNU:AEFR의 철학에서 "동기"는 곧 질서입니다.

    *   **화면 테어링(Tearing) 거부**: 메인 스레드는 Update 단계에서 연산 결과를 동기적으로 기다립니다. 이는 매 프레임마다 스켈레톤의 위치, 표정, 물리 효과가 공간적으로 완벽하게 정렬되도록 보장하기 위함입니다 (캐릭터 일러스트가 이리저리 날아다니는 것은 원치 않으시겠죠).
    *   **작업 훔치기(Work Stealing) 메커니즘**: Rayon의 하드코어한 구현 덕분에 코어 수가 적은 극단적인 상황(예: 듀얼 코어 머신)에서는 메인 스레드가 직접 연산에 참여하여 연산 능력 활용률이 100%의 물리적 한계에 도달하도록 합니다.
    *   **스케줄러 경고**: 연산 집약적인 작업에서 `std::thread::spawn`의 사용을 금지합니다. 모든 병렬 작업은 `AefrScheduler`를 통해 디스패치되어야 합니다. 이를 위반하면 시스템 스케줄링이 무질서한 경쟁에 빠지게 되며, 이는 GNU:AEFR의 Rust 미학을 배신하는 행위입니다.

### GUI 개선 규칙
*   UI 컴포넌트는 `egui`의 **즉시 모드(Immediate Mode)** 철학을 엄격하게 따라야 하며, `React` 스타일의 제안은 받아들이지 않습니다.
