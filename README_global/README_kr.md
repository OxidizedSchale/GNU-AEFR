> GNA:AEFR은 현재 모든 사람을 만족시키기 위해 설계된 복잡한 애플리케이션이 아닙니다. 아직 완성 버전에 도달하지 않았기 때문입니다. 이것은 순수 Rust로 구성된 고성능 에디터의 인스턴스일 뿐입니다. **컴퓨터의 세계에서 가장 짧은 경로가 언제나 무적입니다!** 현재의 호환성과 편의성을 추구한다면 AA를 사용하십시오. 만약 극한의 자유와 성능, 그리고 커뮤니티 중심의 유지보수를 원하신다면 **GNA:AEFR**에 오신 것을 환영합니다.

*   **비공식 팬 메이드**: 순수 Rust로 제작된 고성능, 멀티 플랫폼, 멀티 스레드 기반의 《블루 아카이브》 2차 창작 에디터입니다.

*   **전 플랫폼 제패**: GNU/Linux, Android, macOS, Windows를 네이티브로 지원합니다.

## 📖 명령어 참조 매뉴얼

### 1. 장면 및 비주얼

*   **배경 로드**
    *   **명령어**: `BG <image_path>`
    *   **설명**: 배경 이미지를 즉시 교체합니다. `.jpg`, `.png`, `.webp`를 지원합니다.
    *   **예시**: `BG C:\Assets\BlueArchive\BG_Classroom.png`

*   **캐릭터 장착**
    *   **명령어**: `LOAD <slot_ID> <.atlas_path>`
    *   **설명**: 캐릭터를 `0`번부터 `4`번까지 총 5개의 슬롯에 로드합니다. 자동 배치를 지원하며, 로드 성공 시 콘솔에 해당 캐릭터가 사용 가능한 애니메이션 목록이 출력됩니다.
    *   **예시**: `LOAD 0 D:\Assets\Shiroko\Shiroko_Home.atlas`

### 2. 동작 및 연출

*   **동작 전환**
    *   **명령어**: `ANIM <slot_ID> <animation_name> [loop: true/false]`
    *   **설명**: `true`는 반복 재생, `false`는 1회 재생입니다. 동작 이름은 콘솔에 출력된 목록과 정확히 일치해야 합니다.
    *   **예시**:
        ```bash
        ANIM 0 Start_Idle_01 true    # 시로코의 대기 동작 반복 재생
        ANIM 1 Attack_Normal false   # 1번 슬롯 캐릭터가 1회 공격
        ```

### 3. 스크립트 및 대화

*   **대화 전송**
    *   **명령어**: `TALK <name>|<affiliation>|<content>`
    *   **설명**: 표준 키보토스 스타일의 대화창을 렌더링합니다. 타자기 효과를 지원하며 클릭 시 스킵 가능합니다. **매개변수는 반드시 수직 바 `|`로 구분해야 합니다.**
    *   **예시**:
        ```bash
        TALK 스나오오카미 시로코|대책위원회|선생님, 은행 털러 갈 거야?
        TALK 아로나|싯딤의 상자|선생님, 업무 시간에 땡땡이치지 마세요!
        ```

### 4. 오디오 시스템

*   **BGM 재생**
    *   **명령어**: `BGM <audio_path>`
    *   **설명**: 배경 음악을 비동기로 로드하고 끊김 없이 전환하여 재생합니다.
    *   **예시**: `BGM D:\Music\Unwelcome_School.mp3`

*   **음악 정지**
    *   **명령어**: `STOP`
    *   **설명**: 현재 재생 중인 BGM을 즉시 정지합니다.

### 기술 스택의 순수성
본 프로젝트는 핵심 비즈니스 로직과 아키텍처의 100%를 **Rust**로 구현하는 것을 원칙으로 합니다.

*   **C++ 인터옵 거부**: 아키텍처의 순수성과 유지보수성을 위해 C++ 런타임이나 프레임워크(Qt, Unity, Unreal 등)와의 복잡한 상호작용(JNI/복잡한 FFI)을 도입하는 PR은 원칙적으로 거부합니다.
*   **예외 사항**: 기존의 완성된 저수준 C 라이브러리(그래픽, 오디오, 파일 시스템 기초 라이브러리)에 접근하기 위해 Rust 세이프 래퍼를 작성하는 것은 허용됩니다. 이 경우:
    *   커뮤니티에서 유지보수되는 성숙한 `-sys` 바인딩 라이브러리를 우선적으로 사용해야 합니다.
    *   FFI 호출을 위해 `unsafe` 코드를 작성해야 한다면, 아래의 `unsafe` 규범을 엄격히 준수하고 그 불가대체성을 증명해야 합니다.
    *   최종적인 목표는 모든 `unsafe` 호출을 안전한 Rust API 내에 캡슐화하여 상위 애플리케이션 계층에 완전히 투명하게 만드는 것입니다.

### unsafe 규범: 안전 위의 성능이라는 칼날
> `unsafe` 키워드는 컴파일러의 안전 규칙 밖에서 필요한 저수준 작업을 수행하기 위해 Rust가 개발자에게 부여한 권한입니다. 본 프로젝트는 이를 "안전 위의 성능이라는 칼날"로 간주하며, 그 사용 원칙은 "불필요하면 사용하지 말고, 사용할 때는 만반의 준비를 갖추는 것"입니다.

**핵심 원칙: 필요성 증명**

모든 `unsafe` 코드 블록은 안전한 Rust 코드로는 실현 불가능한 정당한 사유가 있어야 합니다. PR 제출 시 `unsafe` 블록 상단에 주석 형태로 간결한 "필요성 증명"을 제공해야 하며, 내용은 최소한 다음을 포함해야 합니다:

1.  **사유**: 왜 `unsafe`를 사용해야 하는지 명시 (예: 특정 C FFI 함수 호출, 결정론적 메모리 레이아웃 변환, 자기 참조 구조체 구현 등).
2.  **불가대체성**: 왜 표준 라이브러리나 커뮤니티 라이브러리로 동일한 기능을 구현할 수 없는지 논증.
3.  **안전 경계**: 해당 `unsafe` 블록이 유지할 것을 약속하는 불변 조건(Invariants)을 명확히 정의. 즉, 개발자로서 어떤 조건이 성립함을 컴파일러에게 "약속"하여 해당 코드가 논리적으로 안전한지 명시하십시오.

*구체적 요구사항: 문서화 주석*
*   모든 `unsafe` 함수, 메서드 또는 코드 블록에는 앞부분에 Safety 주석을 붙여야 합니다.

### 신사적인 스케줄러: 연산 계층에서의 계급 격리
> GNA:AEFR v0.8+부터는 자체 개발한 "신사적인 스케줄러"를 도입했습니다. 우리는 운영체제의 기본 스케줄링을 신뢰하지 않습니다. 운영체제는 흔히 말하는 "공정성"을 위해 실시간 렌더링의 확정성을 희생하곤 하기 때문입니다.

*   **핵심 규범: N-2 전략**
    *   GNA:AEFR은 CPU를 집어삼키는 탐욕스러운 맹수가 되기를 거부합니다. 스케줄러는 물리 코어 수 `N`을 자동으로 감지하고, `N-2`개의 코어를 계산 전용 구역으로 강제 격리합니다(코어/스레드가 2개뿐인 경우 AEFR이 1개를 점유합니다).
    *   **1개 코어** UI/Render(Main Thread) 전용 예약: 계산량이 폭증하더라도 렌더링 프레임레이트를 매끄럽게 유지합니다.
    *   **1개 코어** OS/Audio(Backstage) 전용 예약: 저수준 시스템의 "완충 공간"으로서, CPU 풀 로드로 인해 배경 음악에 노이즈나 끊김이 발생하는 것을 철저히 차단합니다.
    *   나머지 코어는 **GNA:AEFR** Worker에 할당: Rayon 스레드 풀을 이용하여 Spine 골격 스키닝 및 물리 연산을 수행합니다.

*   **왜 "동기 블로킹 모델"을 고집하는가?**
    > 웹 개발에서 흔히 쓰이는 저렴한 "비동기 비차단(non-blocking async)" 개념을 우리에게 강요하지 마십시오. GNA:AEFR의 철학에서 "동기"는 곧 "질서"입니다.

    *   **화면 찢어짐(Tearing) 거부**: 메인 스레드는 Update 단계에서 계산 결과를 동기적으로 기다립니다. 이는 매 프레임의 골격 위치, 표정, 물리 효과가 공간상에서 완벽하게 일치함을 보장하기 위함입니다.
    *   **워크 스틸링(Work Stealing) 메커니즘**: Rayon의 하드코어한 구현 덕분에 코어 수가 적은 극한의 상황(예: 듀얼 코어 기기)에서도 메인 스레드가 직접 계산에 참여하여 연산 능력 활용도를 물리적 한계인 100%까지 끌어올립니다.
    *   **스케줄러 경고**: 계산 집약적 작업에서 `std::thread::spawn`을 사용하는 것을 금지합니다. 모든 병렬 작업은 반드시 `AefrScheduler`를 통해 배포되어야 합니다. 이를 위반하면 시스템 스케줄링이 무질서한 경쟁 상태에 빠지게 되며, 이는 GNA:AEFR의 Rust 미학에 대한 배신입니다.

### GUI 개선 규칙
*   UI 컴포넌트는 `egui`의 **즉시 모드(Immediate Mode)** 철학을 따라야 합니다. `React` 스타일의 선언적 상태 관리 제안은 수용하지 않습니다.
